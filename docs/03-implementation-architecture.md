# 实现方案与总体架构

## 1. 架构目标

MVP 采用前后端分离和后端模块化单体。目标是在较短周期内交付可试点系统，同时保持生产级边界：可审计、可追溯、可扩展、可降级、可接入统一认证。

## 4. 核心链路

### 4.1 ReAct 对话链路

1. 前端调用会话 API 或 SSE 接口。
2. 后端创建 `AgentRun`。
3. `UrbanManagementAgent` 识别意图、风险和可用工具。
4. ReActAgent 执行推理、工具调用、观察、继续推理。
5. 工具调用前后写入 `ToolCall`。
6. 对政策、法规、执法流程、处罚依据等咨询，回答前必须具备可引用知识来源；未检索到来源时返回“资料不足”提示，不调用模型补写正式答复。
7. 最终回答写入 `Message`，引用来源写入消息结构化字段。
8. 前端展示答案、引用来源、工具摘要和风险等级。

### 4.2 RAG 知识链路

1. 管理员上传文档。
2. 后端抽取元数据、解析正文、分块。
3. 调用 embedding 模型生成向量。
4. 写入 `knowledge_document`、`knowledge_chunk` 和 `knowledge_chunk_embedding`；PostgreSQL profile 下同步 pgvector 原生列。
5. 查询时按租户、区域、密级、文档状态过滤。
6. 向量召回后返回来源片段。
7. 回答必须引用文件名、章节、发布日期或生效日期。
8. 对必须引用来源的问题，若 TopK 召回为空，后端返回统一无来源提示，避免模型编造政策、条款或流程。

### 4.3 智能问数链路

1. 用户输入自然语言问题。
2. 系统解析指标、维度、时间、区域和排序要求。
3. 查询授权数据目录。
4. LLM 生成候选 SQL。
5. SQL AST 校验。
6. 权限改写和字段脱敏。
7. 使用只读账号执行。
8. 返回结果、口径说明、数据来源和审计摘要。

### 4.4 风险与审核链路

1. 风险规则和模型共同评估问题风险。
2. 低风险直接回答。
3. 中风险回答时增加适用条件和风险提示。
4. 高风险进入事实追问或 `pending_legal_review`。
5. 法制审核人员通过、修订、驳回或要求补充事实。
6. 审核结果写入审计并返回给用户。

## 5. 部署方式

MVP 推荐：

- 前端：Vite 构建后由 Nginx 托管。
- 后端：Spring Boot jar 独立部署。
- 数据库：PostgreSQL 开启 pgvector 扩展；本地 H2 仅用于开发和自动化测试，不执行 pgvector 专属迁移。
- 反向代理：Nginx 将 `/api` 和 `/sse` 转发到后端。
- 日志：后端输出 JSON 结构化日志。

后续可容器化部署，按模块拆分服务。

## 6. 关键设计决策

| 决策 | 选择 | 理由 |
| --- | --- | --- |
| MVP 架构 | 模块化单体 | 交付快，边界清晰，后续可拆服务 |
| 前端 | Vue 3 + Vite | 轻量、开发快、适合业务系统 |
| 向量库 | pgvector | 与 PostgreSQL 统一部署，降低 MVP 运维复杂度 |
| 问数 | 完整 NL2SQL + 强校验 | 满足能力要求，同时控制 SQL 风险 |
| 权限 | 内置权限 + OIDC 预留 | 试点可跑，生产可接统一认证 |
| 审计 | 全链路记录 | 满足政务合规和问题追溯 |
